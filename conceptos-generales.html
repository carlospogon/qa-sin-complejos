Conceptos Generales

Las pruebas ¿Cuántas son necesarias?

Una sencilla pregunta, pero que es capaz de desatar todo un vendaval de pensamientos, reflexiones y respuestas: ¿Cuántas pruebas son suficientes para calificar una versión de nuestro software?

Un vendabal de respuestas


Para responder a esta pregunta debemos analizar varios factores como son, por ejemplo: el tipo de software, el objetivo final, el público al que va dirigido… Por ejemplo, sería sencillo la rigurosidad en resultados de motores de búsqueda convencionales, que en aplicaciones internas o para un público muy reducido.

No obstante, sin importar la tipología del producto final, la pregunta de cuántas pruebas es suficientes puede ser difícil de responder en términos concretos. Un mejor enfoque es proporcionar consideraciones o reglas generales que se puedan usar para definir un proceso de calificación y una estrategia de prueba que mejor se adapte al caso en cuestión.

Posibles consejos útiles

  • Documentar nuestro proceso o estrategia.
  • Repositorio sólido de pruebas unitarias.
  • No escatimar en las pruebas de integración.
  • Realice pruebas de extremo a extremo (end-to-end) en los journeys críticos de los usuarios.
  • Comprender e implementar los otros niveles de prueba.
  • Comprensión profunda de nuestra cobertura de código y funcionalidad.

Documentar nuestro proceso o estrategia


Un aspecto importante a tener en cuenta si ya estmos probando nuestro producto es documentar todo el proceso. Esto es esencial para poder repetir la prueba para una versión posterior y analizarla para mejorar aún más. Si encontramos ante el desafío que este sea nuestro primer lanzamiento, una buena idea sería tener un plan o estrategia de prueba por escrito.

De hecho, tener un plan o estrategia de prueba por escrito es algo que debería acompañar a cualquier diseño de producto.

Repositorio sólido de pruebas unitarias


Un excelente lugar para comenzar es escribir pruebas unitarias que acompañen al código. Las pruebas unitarias prueban el código tal como está escrito en el nivel de unidad funcional.

En muchas empresas, incluida Google, existen prácticas recomendadas de exigir cualquier cambio de código para que se aprueben los casos de prueba de unidad correspondientes. A medida que la base de código se expande, tener un cuerpo de tales pruebas que se ejecute antes de que se envíe el código es una parte importante para detectar errores antes de que entren en la base de código. Esto ahorra tiempo más adelante tanto en la escritura de pruebas de integración, depuración y verificación de correcciones al código existente.

No escatimar en las pruebas de integración.


Conforme nuestra base de código se expande y llega a un punto en el que hay un número de unidades funcionales disponibles para probar como grupo, es hora de tener una base sólida de pruebas de integración. Una prueba de integración toma un pequeño grupo de unidades, a menudo solo dos unidades, y prueba su comportamiento como un todo, verificando que funcionan juntas de manera coherente.

A menudo, los desarrolladores piensan que las pruebas de integración pueden perder prioridad o incluso omitirse en favor de pruebas completas de un extremo a otro. Después de todo, este último realmente prueba el producto como lo ejercitaría el usuario. Sin embargo, tener un conjunto completo de pruebas de integración es tan importante como tener una base sólida de pruebas unitarias

La razón radica en el hecho de que las pruebas de integración tienen menos dependencias que las pruebas completas de extremo a extremo. Como resultado, las pruebas de integración, con entornos más pequeños para generar, serán más rápidas y confiables que las pruebas completas de extremo a extremo con su conjunto completo de dependencias
